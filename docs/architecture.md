# Sapling Backend Architecture

The backend ingests student documents, runs Gemini-powered classification, summarization, and assignment/concept extraction, and serves a knowledge-graph-backed tutor and quiz engine to the React frontend.

## High-level shape

A single FastAPI app (`backend/main.py:87`) mounts every router under `/api/<name>`. Supabase is both the datastore and the source of user identity â€” accessed exclusively through a thin PostgREST wrapper (`backend/db/connection.py:102`) using a service-role key. Gemini is still the live runtime path for most LLM work (via `services/gemini_service.py`), but the migration to Pydantic AI agents under `backend/agents/` has begun: `pydantic-ai-slim[google]` is now a dependency, and the `/api/notes` and `/api/quiz` agent actions already route through agents rather than `gemini_service`. The React app in `frontend/` is the only consumer of these endpoints; there is no public/third-party API surface.

## Request paths

- **Document upload** â€” `backend/routes/documents.py:603` `upload_document` runs sequentially: validate â†’ `extraction_service.extract_text_from_file` â†’ `_process_document` (one `call_gemini_json` for category/summary/concepts/assignments) â†’ optional `save_assignments_to_db` (`backend/services/calendar_service.py:111`) for syllabi â†’ optional `apply_graph_update` for syllabus/assignment concepts â†’ insert `documents` row â†’ invalidate `study_guides` cache â†’ `check_achievements("documents_uploaded")`.
- **Chat with tutor** â€” `backend/routes/learn.py:582` `chat` rebuilds the system prompt via `build_system_prompt` (`backend/routes/learn.py:261`) using the live graph + course documents + cached `course_context`, calls `call_gemini_multiturn`, splits out `<graph_update>` via `extract_graph_update`, persists the assistant message, then calls `apply_graph_update` which lazy-imports `update_course_context` for any touched course.
- **Quiz generation** â€” `backend/routes/quiz.py:300` `generate_quiz` loads the target node + prior `quiz_context` and runs the Pydantic AI `quiz_agent` (`backend/agents/quiz.py`); on agent failure it falls back to `_legacy_generate_quiz`, which fills `prompts/quiz_generation.txt`, appends class-wide misconceptions and weak areas from `course_context_service.get_course_context` via `prompt += ...`, and calls `call_gemini_json`. Result is stored in `quiz_attempts`.
- **Study guide** â€” `backend/routes/study_guide.py:21` `_generate_and_insert` fetches the exam row + all offering `documents`, concatenates `summary` + `concept_notes` into a context block, calls `call_gemini_json`, and inserts into `study_guides`. The `/guide` GET serves cache-first; `upload_document` invalidates by deleting that user+offering's rows.
- **Calendar / syllabus** â€” covered by the syllabus branch of `upload_document` above (`save_assignments_to_db` deduplicates by trimmed-title + calendar-day). The standalone `backend/services/calendar_service.py:168` `process_and_save_syllabus` exists for direct OCRâ†’Geminiâ†’DB use but is not currently wired to a route.

## LLM seam (current)

Most LLM calls in the codebase still route through `backend/services/gemini_service.py`, which holds a single module-level `genai.Client` pointed at `gemini-2.5-flash`. The four public entry points are `call_gemini` (`:64`, plain text), `call_gemini_multiturn` (`:90`, native chat history with system instruction), `call_gemini_json` (`:135`, JSON-mode + tolerant `_extract_json` fallback), and `extract_graph_update` (`:147`, parses the `<graph_update>` block out of tutor replies). This is the legacy seam, and the migration off it is underway: new LLM-driven work lands as Pydantic AI agents under `backend/agents/` (model slots in `agents/_providers.py`), replacing call sites incrementally (see `docs/decisions/`). That directory now exists and `pydantic-ai-slim[google]` is in `requirements.txt`; the `/api/notes` agent actions (`note_summary`/`note_concepts`/`note_chat`) and quiz generation (`agents/quiz.py`, with a `gemini_service` fallback) already run on agents, while document upload, tutor chat, study guides, and social still call `gemini_service` directly.

## Data layer

All Supabase access goes through `backend/db/connection.py:102` `table()`, which returns a `SupabaseTable` wrapping `select/insert/update/upsert/delete` over PostgREST with a single shared `httpx.Client`. There is no ORM. Schema is now delivered as ordered raw-DDL migrations under `backend/db/migrations/` (`0001`â€“`0028`), applied by a minimal psycopg runner (`backend/db/migrate.py`) over the Supabase *direct* connection string â€” the one sanctioned exception to the `table()`-only convention, since PostgREST cannot execute DDL. Staging is populated by an idempotent fake dataset in `backend/db/seed_staging.py`. The client uses `SUPABASE_SERVICE_KEY` (service role), so PostgREST runs without RLS enforcement; ownership is enforced in route code by filtering every query on `user_id` and validating via `_validate_user`.

## Data model

The schema is organized into eight bounded domains â€” Academics, Identity, Gradebook, Knowledge Graph, Study, Analytics, Ops, and Gradescope (migrations `0019`â€“`0028`). The central change is the catalog/offering/term split: an abstract `courses` catalog, per-term `course_offerings` (FK to `courses` + `terms`), and `terms`; the old `user_courses` table became `enrollments` keyed on `offering_id`. Different domains key on different levels of this split, bridged by `services/academics.py` (`current_term`, `resolve_offering`, `offering_course_id`): the knowledge graph stays on the **abstract `course_id`** so mastery is cumulative across terms, the gradebook keys on **`enrollment_id`**, and study artifacts + class analytics key on **`offering_id`**. The HTTP API boundary still speaks the abstract `course_id`; the resolver translates inward.

Identity is likewise split: `users` is slimmed to account state, with a 1:1 `user_profiles` table holding the encrypted name/profile fields; `services/profiles.py` provides decrypted display-name reads. The gradebook is enrollment-keyed (`gradebook_categories` with drop-lowest, enrollment-keyed `assignments`, plus curve policy on `enrollments`), and class analytics moved to `offering_concept_stats` / `offering_summary` (was `course_concept_stats` / `course_summary`). Knowledge-graph mastery history is now an append-only `node_mastery_events` table (replacing the old `graph_nodes.mastery_events` JSONB column), with UNIQUE-backed upserts for node/edge dedup. Ops tables (`feedback`, `issue_reports`) carry text/uuid PKs and FKs consistent with the rest of the schema.

## Background work

Almost everything currently runs inline on the request path. The one true background task is in `backend/routes/quiz.py:483`, where FastAPI `background_tasks.add_task(_update_context, ...)` defers the post-quiz context regeneration. `backend/services/course_context_service.py:109` `update_course_context` is called inline from `apply_graph_update` and from course add/delete, but its expensive Gemini summary is hash-gated â€” it only re-runs `_generate_summary_with_gemini` when the stats hash changes. It resolves the touched course to an `offering_id` and upserts into `offering_concept_stats` / `offering_summary`. Achievement checks (`check_achievements`) are also inline and wrapped in best-effort `try/except`.

## Known sharp edges

- `upload_document` is fully sequential: text extraction, the classify/summarize/extract LLM call, optional assignment save, optional graph update, the documents insert, study-guide cache invalidation, and achievement check all run in series on the request thread. Wall-clock â‰ˆ extraction time + LLM time.
- `apply_graph_update` (`backend/services/graph_service.py:461`) is called from three routers with different shapes: `backend/routes/documents.py` passes `{"new_nodes": [...]}` only, `backend/routes/learn.py` passes the full `{new_nodes, updated_nodes, new_edges, recommended_next}` parsed out of the tutor's `<graph_update>` tag, and quiz scoring (`backend/routes/quiz.py`, the `submit_quiz` path) now routes its mastery delta through it too â€” it no longer writes `graph_nodes` directly, instead appending a `node_mastery_events` row via `apply_graph_update`.
- The legacy quiz path (`_legacy_generate_quiz`) manually concatenates course-wide misconceptions onto the prompt string (`backend/routes/quiz.py:241`, `prompt += ...`) after the template substitution, so the prompt template alone does not reflect the final prompt sent to Gemini. The primary path uses `agents/quiz.py`, whose tools pull misconceptions/weak-areas themselves.
- `apply_graph_update` lazy-imports `services.course_context_service` to avoid a circular import; that import path also swallows exceptions so a failing aggregation will not surface to the caller.
- `course_context_service.update_course_context` resolves the course to an offering and writes to `offering_concept_stats` and `offering_summary` synchronously inside the request â€” a chat turn that touches several courses incurs that aggregation cost serially before returning.
- Document-upload concept population only runs for `category in ("syllabus", "assignment")`; lecture notes, slides, readings, and study guides do NOT auto-populate the graph from upload â€” that path is the separate `/scan-concepts` endpoints (`backend/routes/documents.py:1106`, `:1149`).
- Sessions are lazy: `start_session` does not persist anything; the row is materialized inside `_consume_pending` on the first `/chat` call, so a session that's started and abandoned leaves no DB trace.
